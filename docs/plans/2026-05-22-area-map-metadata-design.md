# 区域地图元数据设计

## 背景

预约端已经有室内导航图，学生可以先按 A 楼、A/B 连廊、B 楼以及后续 C 楼、C/D 连廊、D 楼和楼层定位区域，再进入具体桌椅座位选择。当前地图归类依赖区域名称、楼层和描述中的关键词推断，演示方便，但长期运行时不稳定：管理员改名、英文/中文混用或新增公共区域后，地图可能把区域放错楼栋。

## 方案选择

本阶段采用结构化区域元数据方案，在 `areas` 表上新增轻量字段，而不是引入 GIS、CAD、独立地图服务或完整拖拽地图编辑器。

新增字段：

- `buildingCode`: 楼栋/空间分区，允许 `A`、`B`、`C`、`D`、`CONNECTOR`、`CONNECTOR_AB`、`CONNECTOR_CD`，其中 `CONNECTOR` 兼容表示 A/B 连廊。
- `floorCode`: 地图展示楼层，例如 `1F`、`2F`。
- `areaType`: 区域类型，允许 `STUDY_ROOM`、`HALL`、`CORRIDOR`、`CONNECTOR`。
- `mapX`、`mapY`: 地图内百分比坐标，范围 `0-100`，本阶段主要用于排序和后续可视化扩展。

选择这个方案的原因是它能直接稳定预约端地图，同时保持对现有接口、演示数据和同事签到验证工作的影响最小。字段放在 `areas` 上也符合当前“区域是学生进入座位图前的空间入口”的建模方式。

## 数据和 API

后端通过 Flyway `V16__add_area_map_metadata.sql` 增量添加字段，并为已有 demo 区域回填：

- `Library Area A` -> `A / 1F / STUDY_ROOM`
- `Teaching Building Area B` -> `B / 2F / STUDY_ROOM`

`Area`、`AreaResponse`、`CreateAreaRequest`、`UpdateAreaRequest` 和 `AreaMapper` 同步包含这些字段。`AreaService` 负责规范化：

- `buildingCode` 和 `areaType` 接收小写或带空格输入，但持久化为大写枚举字符串。
- `floorCode` 为空时回落到原有 `floor` 字段。
- `mapX` 和 `mapY` 允许为空，但提供时必须在 `0-100` 之间。
- 未填写 `buildingCode` 或 `areaType` 时不阻塞旧数据，前端继续使用兼容推断。

## 前端交互

预约端 `CampusIndoorMap` 优先使用结构化字段：

- 楼栋按 `buildingCode` 分组。
- 楼层按 `floorCode ?? floor` 筛选。
- 区域排序优先 `mapY`、再 `mapX`、最后名称。

如果某个旧区域没有地图元数据，组件继续用原有名称/描述推断，保证现有演示数据和历史数据可用。

管理员区域管理页新增地图配置字段，放在现有区域基础信息表单中：

- 楼栋分区：A 楼、B 楼、C 楼、D 楼、A/B 连廊、C/D 连廊。
- 地图楼层：可与原“楼层”一致，也可用于后续更精细命名。
- 区域类型：自习室、大厅、走廊、连廊。
- 地图坐标 X/Y：百分比位置，非必填。

手机端继续使用同一套响应式页面，新增字段在窄屏下自然单列，不新增移动端专用页面。

## 边界

- 不修改签到验证、WiFi/IP 校验、签到码校验、锁位恢复和后端签到状态机。
- 不实现完整地图拖拽编辑器；这次只把长期稳定需要的数据结构和维护入口打好。
- 不删除原有 `floor` 字段，避免影响开放时段、桌子、座位和已有页面。
- 不强制所有区域必须填写地图元数据，兼容旧数据和分阶段录入。

## 验证

- 后端单元测试覆盖创建/更新区域时字段规范化、默认楼层回落、非法楼栋/类型/坐标拒绝。
- 前端组件测试覆盖结构化 `buildingCode/floorCode` 优先于名称推断。
- 前端页面测试覆盖管理员保存区域时会提交地图元数据。
- 全量运行后端测试、前端测试、lint、build 和 `git diff --check`。
