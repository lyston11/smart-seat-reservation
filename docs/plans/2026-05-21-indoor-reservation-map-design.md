# 预约端室内地图设计

## 背景

预约端需要支持学院公共区域座椅的空间化选择：按楼栋、楼层和 A/B 楼连廊组织区域，学生先像看室内地图一样定位区域，再进入具体桌椅座位选择。

## 方案选择

本阶段采用前端室内示意图方案，不引入 GIS、CAD 或地图引擎，也不新增后端表字段。页面复用现有 `areas`、`seats`、`seat_slots` 数据，先从区域名称、楼层和描述中推断 A 楼、B 楼或连廊，形成可点击的室内导航层。

这个方案的优点是改动小、演示效果直观、不会影响同事正在做的签到验证闭环。后续如果要长期扩展，可以在后端补充 `buildingCode`、`floorCode`、`areaType`、`mapX/mapY` 等字段，当前组件只需要替换推断逻辑即可继续复用。

## 交互设计

- 顶部提供楼层切换，同一页面响应式适配桌面和手机端。
- 地图主体分为 A 楼、A/B 连廊、B 楼三段。
- 每段展示当前楼层下已启用区域，点击区域后同步学生选座页当前区域。
- 下方继续保留现有区域下拉、日期、时间规则、统计和真实桌椅 `SeatMap`。
- 具体桌子的具体座位仍由 `SeatMap` 展示和选择，保留长方形桌、上下座位和锁位状态展示。

## 数据流

`SeatSlotsPage` 加载 `areas` 后传给 `CampusIndoorMap`。地图选择区域时调用现有 `applySelectedArea`，从而复用已有加载座位、生成未开放座位、预约创建和状态校验逻辑。

## 边界

- 本次只做预约端 UI，不改签到验证、WiFi/IP 校验、签到码校验和后端状态机。
- 本次不新增数据库迁移，避免和同事后端工作冲突。
- 区域楼栋推断只是演示期兼容层，真实长期配置建议后续由管理员维护结构化楼栋和地图坐标。

## 验证

- 新增 `CampusIndoorMap` 组件测试，覆盖 A/B 楼、连廊、楼层筛选和区域点击。
- 新增学生选座页集成测试，覆盖从地图切换区域后请求对应区域座位。
- 运行前端测试、lint、build，并在必要时保留已有后端测试验证。
