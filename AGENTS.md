# AI 开发规范

本文件是 `smart-seat-reservation` 项目的 AI 协作开发规则。所有团队成员使用 Codex、Cursor、Claude Code、GitHub Copilot、ChatGPT 或其他 AI 工具开发本项目时，都必须让 AI 优先阅读并遵守本文档。

AI 的职责不是“随便生成能跑的代码”，而是以工程化方式协助开发：先理解项目，再小步实现，最后验证和说明。

## 1. 最高优先级规则

- 不允许直接在 `main` 分支开发功能。
- 不允许直接 push 到 `main` 分支。
- 每次开始任务前，必须先同步主分支。
- 每次开始任务前，必须先阅读 `docs/dev-logs/` 下自己和其他成员的开发日志。
- 每个任务必须在自己的功能分支中完成。
- 每次开发结束后，必须更新当前成员自己的开发日志。
- 每次改动必须保持范围清晰，不混入无关修改。
- 提交前必须检查代码、运行项目或说明未验证原因。
- 生成的代码必须符合项目已有结构、命名和风格。
- 不允许提交密钥、密码、Token、私钥、真实数据库地址等敏感信息。

如果当前分支是 `main`，AI 必须先提醒开发者创建或切换到功能分支，不能继续直接开发业务代码。

## 2. 开始任务前必须执行

AI 在修改代码前，应先确认当前仓库状态：

```bash
git status
git branch --show-current
```

如果不在最新主分支基础上开发，应先同步：

```bash
git checkout main
git pull origin main
```

同步后必须阅读开发日志：

```bash
ls docs/dev-logs
```

AI 应读取：

- 当前开发者自己的日志。
- 其他成员最近的日志。
- 与当前任务相关的日志记录。

阅读日志后，AI 应简要说明是否发现重复开发、接口冲突、数据库字段冲突或需要避让的文件。

确认后再创建任务分支：

```bash
git checkout -b feature/your-task-name
```

如果已经有对应任务分支：

```bash
git checkout feature/your-task-name
git pull origin feature/your-task-name
```

推荐分支命名：

```text
feature/student-login
feature/seat-reservation
feature/admin-seat-management
fix/reservation-conflict
docs/update-agent-rules
```

多人并行开发时，建议在分支中加入成员标识：

```text
feature/zhangsan-seat-map
feature/lisi-admin-dashboard
feature/wangwu-checkin-flow
```

## 3. 标准开发流程

AI 每次处理任务时，应遵循以下流程：

```text
理解需求
查看相关代码
确认当前分支
同步 main
读取所有成员开发日志
制定小范围实现计划
创建或切换任务分支
修改代码
补充必要测试或验证方式
运行检查
更新当前成员开发日志
总结改动
提醒提交 PR
```

除非开发者明确要求，否则 AI 不应一开始就大范围重构项目。

## 4. 工程化代码要求

AI 生成代码时必须遵守：

- 模块职责清晰，不把大量逻辑堆在一个文件里。
- 命名清楚，变量、函数、组件、接口名称应表达真实含义。
- 业务规则集中管理，避免到处复制粘贴。
- 接口入参、返回值、错误状态要明确。
- 数据库字段、后端模型、前端类型要尽量保持一致。
- 关键业务流程要处理异常情况。
- 删除、覆盖、释放座位等操作要考虑权限和状态校验。
- 预约冲突、重复预约、超时释放等逻辑必须明确边界。
- 不引入没有必要的新框架、新依赖或复杂抽象。
- 不为了“看起来高级”写难维护的代码。

座位预约系统的关键状态建议统一表达，例如：

```text
空闲
已预约
使用中
已释放
已取消
异常占用
```

如果项目中已经定义了状态枚举或常量，必须复用已有定义，不要另起一套。

## 5. AI 修改代码前要先读项目

AI 不能在不了解项目结构的情况下直接生成代码。至少应查看：

```bash
ls
find . -maxdepth 2 -type f
```

如果项目已有前后端目录、配置文件、路由、数据库模型，应优先遵循已有组织方式。

常见需要查看的文件包括：

```text
README.md
DEVELOPMENT.md
package.json
requirements.txt
pyproject.toml
pom.xml
src/
app/
backend/
frontend/
```

实际文件以项目当前结构为准。

## 6. 开发日志要求

每位成员必须在 `docs/dev-logs/` 下维护自己的开发日志。文件名建议使用姓名或 GitHub 用户名：

```text
docs/dev-logs/zhangsan.md
docs/dev-logs/lisi.md
docs/dev-logs/wangwu.md
```

AI 每次开发前必须读取所有成员的开发日志，重点确认：

- 当前是否有人正在做相同功能。
- 是否有人修改了相同文件或模块。
- 是否存在接口路径、字段命名、数据库结构冲突。
- 是否存在未解决的遗留问题。
- 当前任务应该基于哪个分支或 Issue 开始。

AI 每次开发结束后必须协助更新当前开发者自己的日志，追加记录，不覆盖历史记录。

日志模板：

```md
## 2026-05-13

### 任务
- Issue:
- 分支:
- 目标:

### 本次改动
- 

### 涉及文件
- 

### 验证方式
- 

### 遗留问题
- 

### 对其他成员的影响
- 
```

如果开发者没有说明自己的姓名或日志文件名，AI 应先询问，或建议使用 GitHub 用户名作为日志文件名。

## 7. 禁止行为

AI 不允许执行以下行为：

- 在 `main` 分支直接开发功能。
- 直接 push 到 `main`。
- 未阅读开发日志就开始改代码。
- 开发结束后不更新开发日志。
- 未经确认删除大量文件。
- 未经确认重写项目结构。
- 未经确认更换技术栈。
- 未经确认修改与当前任务无关的功能。
- 使用真实账号密码、Token 或私钥写入代码。
- 生成无法解释、无法维护的大段代码。
- 跳过验证并声称功能已经完成。
- 为了解决冲突直接覆盖其他成员代码。

如果必须执行高风险操作，AI 应先向开发者说明影响并等待确认。

## 8. 提交前检查

AI 完成修改后，应提醒开发者至少执行：

```bash
git status
git diff
```

根据项目技术栈，运行相应检查。例如：

```bash
npm install
npm run lint
npm run test
npm run build
```

或：

```bash
pip install -r requirements.txt
pytest
```

如果项目暂时没有测试，也应进行手动验证，并在 PR 中说明验证方式。

提交前还应确认：

- 当前分支不是 `main`。
- 本次任务对应的开发日志已经更新。
- 没有把其他成员日志中的未完成工作误删或覆盖。

## 9. 提交与推送规则

提交信息使用：

```text
类型: 简短说明
```

常用类型：

```text
feat: 新功能
fix: 修复问题
docs: 文档修改
style: 样式调整
refactor: 代码重构
test: 测试相关
chore: 配置或杂项
```

示例：

```bash
git add .
git commit -m "feat: add seat reservation flow"
git push origin feature/seat-reservation
```

禁止：

```bash
git push origin main
```

推送功能分支后，应在 GitHub 创建 Pull Request，由至少一名其他成员 Review 后再合并。

## 10. Pull Request 要求

AI 应协助开发者整理 PR 描述：

```md
## 改动内容
- 

## 验证方式
- 

## 影响范围
- 

## 关联 Issue
Closes #
```

PR 应保持小而清晰。一个 PR 只解决一个主要问题。

如果本次改动影响接口、数据库、路由、权限或部署方式，必须在 PR 描述中明确说明。

PR 描述中应说明开发日志是否已更新。

## 11. 冲突处理规则

遇到代码冲突时，AI 应先解释冲突原因，再协助解决。

解决冲突时必须遵守：

- 不直接删除其他成员代码。
- 不使用强制覆盖来掩盖冲突。
- 保留双方有效逻辑。
- 冲突解决后必须运行验证。
- 不确定时让开发者联系相关成员确认。

禁止随意使用：

```bash
git reset --hard
git push --force
```

除非开发者非常明确地要求，并已理解后果。

## 12. 面向本项目的业务开发要求

本项目是学院公共区域学习座位预约系统。AI 实现功能时，要围绕以下核心业务：

- 学生查看座位状态。
- 学生预约空闲座位。
- 系统防止重复预约和时间冲突。
- 预约成功后生成签到凭证。
- 学生到场扫码签到。
- 使用结束后签退释放座位。
- 超时未签到自动释放座位。
- 管理员维护区域、座位、开放时间和预约规则。
- 管理员查看实时占用情况。
- 管理员手动释放异常占用座位。

涉及这些核心流程时，AI 必须优先保证状态流转正确，而不是只完成页面或接口表面效果。

## 13. AI 回复格式建议

AI 完成任务后，应向开发者说明：

```text
已完成：
- 

修改文件：
- 

验证：
- 

开发日志：
- 

注意事项：
- 
```

如果未能验证，必须明确写出：

```text
未验证原因：
- 
```

不能把未验证的功能描述成“已完全完成”。

## 14. 给 AI 的默认工作指令

团队成员可以在每次新会话开始时把下面这段话发给 AI：

```text
请先阅读本仓库的 AGENTS.md、README.md、DEVELOPMENT.md，以及 docs/dev-logs/ 下所有成员最近的开发日志。
你必须遵守项目协作规范：
1. 不直接在 main 分支开发
2. 不直接 push 到 main
3. 每个任务使用独立功能分支
4. 先理解项目结构，再进行工程化实现
5. 改动后给出验证方式
6. 不做与当前任务无关的大范围重构
7. 开发前先阅读自己和其他成员开发日志
8. 开发结束后更新我的开发日志

当前任务是：请根据我接下来描述的 Issue 完成开发。
```

本文件的目标是让每个人使用 AI 时都能保持一致的工程质量和协作秩序。
